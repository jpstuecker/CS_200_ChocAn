Use Case Diagram

Use Case Descriptions

  • Use Case: Validate Membership
    Context: Provider validates a customer's ChocAn subscription
    Actor(s): Provider
    Main Success Scenario:
    1. A member card is slid through the card reader on a terminal, then the terminal dials the ChocAn data center.
    2. The ChocAn Data Center computer verifies the member number.
    2.1 If valid, the word "Validated" appears on the one line display.
    3. The provider has the option to bill ChocAn for a health care service provided.
    Extensions:
    2a. The card number is invalid.
    2a.1. The word "Invalid" is displayed on the one line display indicating the number is invalid.
    2a.2. The use case continues at step 1.
    2b. The member is suspended.
    2b.1. The word "Suspended" is displayed on the one line display indicating the member has not paid fees for at least one month.
    2b.2. The use case continues at step 1.

  • Use Case: Email Member Report
    Context: Sends emails to members detailing the week's services
    Actor(s): Clock, Manager
    Main Success Scenario:
    1. The system selects a member.
    2. The system gathers all reports made within the past 7 days for that member.
    3. The system creates a list of all the reports to be sent to the member.
    4. The system adds the members information on the report.(name, number, street address, city, state, ZIP)
    5. The system composes an email with the report file attached to it and sends it to the corresponding member.
    6. The system reiterates selecting a new member until all members serviced in the week have received emails.
    Extensions:
    5. The email fails to send.
    5a.1. The system logs the error.

  • Use Case: Email Provider Report
    Context: Sends emails to providers detailing the week's services
    Actor(s): Clock, Manager, Provider
    Main Success Scenario:
    1. The system selects a provider.
    2. The system gathers the providers information.(name, number,street address, city, state, ZIP)
    3. The system collects any and all service reports performed by the provider in the past 7 days.
    4. The system counts the number of consultations that the provider completed in the week.
    5. The system totals the fees for the week.
    6. The system compiles a file with all the information gathered in steps 8-11.
    7. The system sends an email to the provider with this file attached.
    8. The system selects a new provider and repeats the process until all providers have received an email report for the week.
    Extensions:
    7. The email fails to send.
    7a.1. The system logs the error.

  • Use Case: Email Summary Report
    Context: Sends emails to the manager for accounts payable with a summary report
    Actor(s): Clock, Manager
    Main Success Scenario:
    1. For each provider to be paid, the system writes the number of consultations performed by each provider and the total fee owed to the provider for that week to a report.
    2. The system also calculates and lists the total number of providers who provided services, the total number of consultations, and the total fees owed and writes the totals to the report.
    3. The report is sent to the manager of accounts payable.
    Extensions:
    3. The email fails to send.
    3a.1. The system logs the error.

  • Use Case: Create EFT Report
    Context: Generates report to be used by banking software to pay providers electronically
    Actor(s): Clock, Manager
    Main Success Scenario:
    1. The system selects a provider.
    2. If the provider was billed in the last week, the system writes the provider name, number, and total amount owed to the provider to an EFT file stored on the disk.
    3. The system selects a new provider and repeats the process until all providers who were billed in the last week are written to the file.
    Extensions:
    N/A

  • Use Case: Update Member Information
    Context: The system adds, removes, or updates a member in the directory.
    Actor(s): Operator
    Main Success Scenario:
    1. The system prompts the operator the option to choose either to add, remove, or update a member.
    2. The operator selects an option
    3. The changes are saved in the system
    4. The system returns to the main menu
    Extensions:
    N/A

  • Use Case: Add Member extends Update Member Information
    Context: The system adds a member in the directory.
    Actor(s): Operator
    Main Success Scenario:
    1. The operator chooses the add option
    2. The system prompts the operator to input the members information (name, number, street address, city, state, zip)
    3. The system adds the data of the member to the database
    4. The system returns to the Update Member Information menu
    Extensions:
    2. The operator enters invalid information or omits information.
    2a.1. The system displays an error message.

  • Use Case: Remove Member extends Update Member Information
    Context: The system removes a member from the directory.
    Actor(s): Operator
    Main Success Scenario:
    1. The operator chooses the remove option
    2. The system displays a list of members that can be removed
    3. The operator selects the member to be removed
    4. The system removes the data of the member from the database
    5. The system returns to the Update Member Information menu
    Extensions:
    N/A

  • Use Case: Update Member extends Update Member Information
    Context: The system updates a member in the directory.
    Actor(s): Operator
    Main Success Scenario:
    1. The operator chooses the update option
    2. The system displays a list of members that can be update
    3. The operator selects the member to be updated
    4. The system then prompts the operator what information the member would like to update (name, number, street address, city, state, zip)
    5. The operator selects the information they want to update.
    6. The operator then inputs the new information
    7. The system updates the data of the member from the database
    8. The system returns to the Update Member Information menu
    Extensions:
    6. The operator enters invalid information or omits information.
    6a.1. The system displays an error message.

  • Use Case: Request Provider Directory
    Context: The provider requests a Provider Directory, which lists the service names and their corresponding service codes and fees.
    Actor(s): Provider
    Main Success Scenario:
    1. The provider requests a Provider Directory.
    2. The system generates a list of service names, codes, and fees.
    3. The system emails the provider the list.
    Extensions:
    3. The email fails to send.
    3a.1. The system returns to provider menu.

  • Use Case: Run Main Accounting Procedure
    Context: Each Friday at midnight, the system runs the main accounting procedure and generates reports for various constituents.
    Actor(s): Clock
    Main Success Scenario:
    1. The system generates and sends a report for each member who received service in the previous week according to the Email Member Report use case.
    2. The system generates and sends a report for each provider who billed ChocAn in the previous week according to the Email Provider Report use case.
    3. The system generates an EFT report for each provider to be paid according to the Create EFT Report use case.
    4. The system generates and sends a summary report to the manager of accounts payable according to the Email Summary Report use case.
    Extensions:
    N/A

  • Use Case: Record Bill for Member
    Context The Provider adds the members bill to the system
    Actor(s): Provider
    Main Success Scenario:
    1. Member gives card to provider to scan.
    2. Provider scans card.
    3. The membership is validated, according to the Validate Membership use case.
    4. The provider scans card again.
    5. The provider enters the date of service into system in this format: MM-DD-YYYY
    6. Provider looks up 6-digit service code and enters it into the system.
    7. System displays the service corresponding to the entered service code and asks the provider to confirm the service.
    8. Provider then can enter an optional comment up to 100 characters.
    9. The system writes a record to the disk the current date and time, date of service, provider number, member number, service code, and comments.
    10. System then shows fee to be paid.
    11. Provider enters the current date and time, the date the service was provided, member name and number, service code, and fee to be paid into a verification form.
    Extensions:
    5a. The provider enters an invalid date.
    5a.1. The system displays an error message.
    6a. The code entered is invalid.
    6a.1. The system displays an error message.
    11a. The provider enters invalid information or omits information.
    11a.1. The system displays an error message.
